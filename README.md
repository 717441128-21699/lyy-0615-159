# HTTP 反向代理负载均衡器

一个功能完整的 Go 语言实现的 HTTP 反向代理 + 负载均衡器。

## 功能特性

- 多种负载均衡策略：轮询（加权）、最少连接、一致性哈希
- 后端健康检查，支持故障自动摘除与恢复
- 连续失败/成功的滞后判定，抵抗网络抖动
- 简单的请求重试机制
- 配置热加载，平滑过渡不中断进行中的请求
- 管理 API 接口，查看状态和手动触发重载

## 核心设计原理

### 1. 健康检查：连续失败/成功的滞后判定

**问题**：网络抖动可能导致健康检查偶发失败，如果一次失败就摘除后端，会造成后端频繁地被摘除和加回（flapping），导致流量震荡和会话丢失。

**解决方案**：采用"滞后判定"（hysteresis）机制，即：

- **故障摘除**：需要连续 `failure_threshold` 次健康检查失败，才将后端标记为不健康并从可用池中摘除。
- **故障恢复**：需要连续 `success_threshold` 次健康检查成功，才将后端重新标记为健康并加回池中。

**实现位置**：[backend/backend.go](file:///d:/trae-bz/TraeProjects/159/backend/backend.go#L78-L118)

```
网络抖动示例（failure_threshold=3, success_threshold=2）：

检查结果:  ✅  ❌  ❌  ✅  ❌  ❌  ❌  ✅  ✅  ✅
状态:      H   H   H   H   H   H   U    U   U   H
          ^                           ^       ^
        初始健康                 3连失败   2连成功
        保持健康                 才摘除     才恢复
```

**为什么有效**：
- 短时网络抖动（1-2次失败）不会触发摘除
- 后端恢复后需要持续稳定一段时间才能重新接收流量
- 阈值越大，抗抖动能力越强，但反应速度越慢，需要根据场景权衡

---

### 2. 一致性哈希：后端增减时最小化会话重映射

**问题**：普通取模哈希（如 `hash(key) % N`）在后端数量 N 变化时，几乎所有 key 的映射都会改变，导致大量会话丢失。

**解决方案**：使用 **一致性哈希环** + **虚拟节点** 技术。

**基本思想**：
- 将后端服务器映射到一个 0 ~ 2³²-1 的哈希环上
- 请求的 key 也哈希到环上，顺时针找到的第一个后端就是目标
- 增加/删除一个后端时，只会影响它在环上相邻的那一小段 key，其余 key 的映射不变

**虚拟节点的作用**：
- 每个真实后端对应 `virtualNodes` 个（默认 150）虚拟节点散列在环上
- 解决"数据倾斜"问题：后端少时分布不均匀
- 后端增减时，重映射均匀地分摊到所有后端上

**实现位置**：[balancer/balancer.go](file:///d:/trae-bz/TraeProjects/159/balancer/balancer.go#L122-L205)

**重映射影响对比**：

| 方案 | N=3 → N=4 时重映射比例 |
|------|----------------------|
| 普通取模哈希 | ~75%（几乎全部改变）|
| 一致性哈希（无虚拟节点） | ~25%（1/N）|
| 一致性哈希（150虚拟节点） | ~25%，且均匀分布 |

**故障时的容错**：
- 选中的后端如果不健康，会顺时针找下一个虚拟节点对应的后端
- 因此一个后端挂掉时，它的流量会分摊到环上的"下一个"后端上

### 3. 配置热加载：不中断进行中请求的平滑过渡

**问题**：配置变更时，如果粗暴地替换内部状态，可能导致正在进行的请求出错（比如正在转发到一个被删除的后端）。

**解决方案**：采用 **原子快照（Atomic Snapshot）** + **引用保留** 模式。

**核心设计**：

1. **不可变快照**：每次配置变更时，构建一个全新的完整状态快照（ServerSnapshot），包含后端池、负载均衡器、健康检查器、代理实例等所有资源。

2. **原子切换**：使用读写锁保护当前使用的快照指针。新请求拿到快照引用后，在整个请求生命周期内都使用这个引用，不受后续配置变更的影响。

3. **优雅回收**：旧快照的资源（如健康检查器）不会立即销毁，而是延迟一段时间（30秒）后再停止，确保所有进行中的请求都已完成。

4. **后端复用**：配置变更时，名字相同的后端对象会被复用，保留其健康状态和连接计数，避免状态重置造成的抖动。

**实现位置**：[server/server.go](file:///d:/trae-bz/TraeProjects/159/server/server.go#L32-L150)

**请求处理流程**：

```
请求到达
   |
   v
获取当前快照（读锁）
   |
   +--- 使用该快照处理请求（整个期间引用不变）
   |
   v
请求完成

（配置变更可能发生在请求处理期间，
 但该请求仍使用旧快照，不受影响）
```

**配置变更流程**：

```
检测到配置变化
   |
   v
构建新快照（新后端池 + 新负载均衡器 + 新健康检查器）
   |   （同名后端对象复用，保留健康状态）
   v
原子切换当前快照指针
   |
   v
旧快照健康检查器延迟 30s 后停止
   |
   v
平滑过渡完成
```

**为什么这样设计**：
- **零中断**：正在进行的请求永远不会因为配置变更而失败
- **一致性**：单个请求的生命周期内使用的是同一套配置，不会前后不一致
- **简单性**：不用维护复杂的引用计数，靠时间窗口保证安全
- **状态保留**：复用同名后端对象，避免健康状态和连接计数被重置

## 管理 API

| 路径 | 方法 | 说明 |
|------|------|------|
| `/health` | GET | 负载均衡器自身健康检查 |
| `/api/status` | GET | 总体状态（后端数、策略等）|
| `/api/backends` | GET | 所有后端的详细状态 |
| `/api/config` | GET | 当前配置 |
| `/api/reload` | POST | 手动触发配置重载 |

## 注意事项

1. 一致性哈希策略下，如果所有后端都不健康，仍会返回 503
2. 重试只对幂等请求安全，非幂等请求（如 POST）需谨慎使用
3. 配置热加载有 2 秒的文件检测间隔，也可通过管理 API 立即触发
